## Node.js architecture — the big picture

Before diving into V8's internals, let's recall the high-level architecture
of Node.js:

<Callout type="insight" title="Node.js = V8 + libuv + core modules">
  Node.js is a runtime that combines several components inside a single dashed boundary:

  - **V8 JS Engine (by Google):** The heart — parses, compiles, and executes JavaScript code
  - **libuv:** The C library that handles async I/O, event loop, thread pool, OS communication
  - **Core modules:** Built-in libraries like `fs`, `http`, `crypto`, `zlib`, and many more — these are the APIs that give JavaScript the power to interact with files, network, and OS

  This episode focuses entirely on what happens **inside V8** — the
  journey your code takes from raw text to actual execution.
</Callout>

## Stage A: Parsing — from code to tokens to AST

When JavaScript code enters V8, the very first thing that happens is
**parsing**. Parsing has two sub-stages: **Lexical Analysis
(Tokenization)** and **Syntax Analysis (AST generation)**.

<Callout type="insight" title="Step 1 — Lexical analysis (tokenization)">
  The main goal of lexical analysis is to break down raw JavaScript code
  into manageable pieces called **tokens**. Each token represents a
  fundamental element of the language.
</Callout>

For example, take this simple line of code:

```js title="input code"
var a = 10;
```

V8's lexer scans this character by character and produces these tokens:

- `var` — keyword
- `a` — identifier
- `=` — operator
- `10` — literal
- `;` — punctuation

```text title="tokenization output"
JavaScript Code: var a = 10;
    ↓
Tokens: [ 'var', 'a', '=', '10', ';' ]
```

**Why tokenization?** It helps V8 read and understand the code by
breaking it into smaller, structured pieces. Without tokenization, V8
would be staring at a wall of characters with no idea where one keyword
ends and another begins. This step is crucial for the next stage —
building the tree.

<Callout type="insight" title="Step 2 — Syntax analysis (building the AST)">
  After tokenization, the flat list of tokens is analyzed according to
  JavaScript's grammar rules and converted into an **Abstract Syntax
  Tree (AST)** — a tree-like data structure that represents the
  hierarchical, syntactic structure of your code.
</Callout>

For `var a = 10;`, the AST looks like this:

```text
VariableDeclaration
├── Identifier (a)
└── Literal (10)
```

Each node in the tree corresponds to a construct in the code:
**VariableDeclaration** represents the statement, **Identifier** is the
variable name `a`, and **Literal** is the value `10`.

For more complex code like this:

```js title="AST Explorer example"
var name = "Node JS";
function sayNamaste() {
  console.log("Namaste World");
}
```

The AST becomes a much larger tree with a **Program** root node
containing a **VariableDeclaration** node and a **FunctionDeclaration**
node, which itself contains an **Identifier** (`sayNamaste`), **params**
(empty array), and a **BlockStatement** body with an
**ExpressionStatement** inside it.

<Callout type="tip" title="Tool">
  You can explore AST structures visually at **astexplorer.net** — paste
  any JavaScript code and see the exact tree V8 would generate.
</Callout>

<Callout type="warning" title="Syntax errors explained">
  When V8 processes tokens one by one during syntax analysis, if it
  encounters an **unexpected token** that doesn't fit JavaScript's
  grammar rules, a **syntax error** occurs. This happens because the AST
  **cannot be generated** if the code structure is invalid. For example,
  `var name =;` would produce "Unexpected token (1:10)" because the
  parser expects a value after `=` but finds `;` instead.
</Callout>

## Stage B: Interpreted vs compiled — and why JS is both

Before understanding V8's execution pipeline, you need to know the two
fundamental types of programming languages:

| Interpreted languages                   | Compiled languages                                  |
| --------------------------------------- | --------------------------------------------------- |
| Executed **line by line** by an interpreter | First **compiled** entirely: high-level code → machine code |
| **Fast initial execution** — no compilation wait | **Slow initial start** (compilation takes time)     |
| Slower runtime performance              | But **very fast execution** once compiled           |
| Easier to debug                         | Harder to debug                                     |
| Example: **Python**                     | Example: **C, C++**                                 |

<Callout type="insight" title="Q: Is JavaScript interpreted or compiled?">
  **Neither purely interpreted nor purely compiled.** JavaScript uses a combination of both techniques — this is called **JIT (Just-In-Time) compilation**.

  - **Interpreter:** JavaScript first uses an interpreter for fast initial execution — start running the script quickly, line by line
  - **Compiler:** V8 then uses JIT compilation to optimize frequently-run code paths into machine code at runtime, just before they're needed — giving compiled-language speed for hot paths

  This is why JavaScript is called **"the ultimate hybrid"** — it gets
  the best of both worlds: interpreted flexibility and compiled speed.
</Callout>

## The V8 execution pipeline — Ignition + TurboFan

This is the core of Episode 08. The whiteboard diagrams on PDF pages 2-3
show the complete V8 pipeline. Here it is as a flow:

- **Source code (your .js file).** Raw JavaScript text enters V8.
- **A. Parsing.** Lexical Analysis: Code → Tokens (tokenization). Syntax Analysis: Tokens → AST (Abstract Syntax Tree).
- **B. Ignition (Interpreter).** Converts AST → **Bytecode** (intermediate representation, lower-level than source code but not yet machine code). Ignition then executes this bytecode **line by line** — this gives you fast initial startup.

  ↘ _While executing, Ignition monitors "hot" code_ — functions that
  are called repeatedly. When code is hot enough, it's sent to
  TurboFan for optimization.

- **C. TurboFan (Optimizing Compiler).** Takes the hot bytecode and compiles it into **optimized machine code** — native CPU instructions. TurboFan makes **assumptions** about types (e.g., "this function always receives numbers") and generates highly efficient code based on those assumptions.

  ↙ _If assumptions are wrong_ (e.g., function was optimized for
  numbers but receives a string), TurboFan **deoptimizes** — the code
  is reverted back to Ignition bytecode for re-interpretation and
  possible re-optimization.

- **D. Execution.** Both bytecode (from Ignition) and optimized machine code (from TurboFan) feed into the execution stage. Your code runs.

<Callout type="insight" title="The hot code example from the whiteboard">
  Imagine a function `sum()` that is called thousands of times:

  - `sum(10, 5)` — Ignition runs it in bytecode
  - `sum(2, 3)` — Ignition runs again, notices it's being called a lot with numbers
  - `sum(6, 7)` — V8 marks this as **"HOT"** → sends to TurboFan
  - TurboFan assumes: "this function always gets numbers" → generates optimized machine code that does fast integer addition
  - `sum("a", "b")` — Oops! Strings, not numbers. The assumption is wrong → **deoptimization** → code goes back to Ignition
</Callout>

<Callout type="tip" title="Developer best practice">
  For optimal performance, pass **consistent types** to your functions.
  If a function is optimized for numeric calculations, avoid passing
  strings. Inconsistent types cause repeated deoptimization →
  reoptimization cycles, which hurt performance.
</Callout>

## Optimization techniques — inline caching & copy elision

V8 uses several clever optimization techniques during bytecode execution
and TurboFan compilation:

**Inline caching.** Speeds up **property access** by caching the results
of lookups. When you access `obj.name`, V8 remembers where `name` was
found in the object's hidden class. Next time, it skips the lookup and
goes directly to the cached location. This makes repeated property
access on same-shaped objects extremely fast.

**Copy elision.** An optimization that **eliminates unnecessary copying
of objects**. When a function returns an object, instead of creating the
object inside the function and then copying it to the caller, V8 can
construct it directly in the caller's memory — avoiding the copy
entirely.

## Garbage collection — memory cleanup behind the scenes

The whiteboard diagrams on PDF page 3 show that **Garbage Collection**
runs **throughout** the entire V8 pipeline — alongside parsing,
interpretation, and compilation. It uses the **Mark and Sweep**
algorithm as its foundation.

<Callout type="insight" title="How mark and sweep works">
  **Mark phase:** V8 starts from "root" references (global scope, call
  stack) and marks every object that is reachable — meaning something
  still points to it.

  **Sweep phase:** V8 goes through the entire heap and frees memory for
  all **unmarked** objects — these are objects that no code references
  anymore.
</Callout>

V8's garbage collector has several specialized sub-systems:

| GC component               | What it does                                                                                                              |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| **Orinoco**                | V8's main garbage collection project — coordinates all GC activities. Runs mostly **concurrently** (in parallel with your JS code) to minimize pauses. |
| **Oil Pan**                | Handles garbage collection for the **C++ objects** that V8 manages internally (DOM nodes in Chrome, internal V8 data structures). |
| **Scavenger**              | Handles the **"young generation"** — newly created objects. Most objects die young (temporary variables, short-lived closures), so Scavenger collects them quickly and frequently using a minor GC cycle. |
| **MCompact** (Mark-Compact) | Handles the **"old generation"** — objects that survived multiple Scavenger cycles. Uses a full mark-sweep-compact cycle: marks reachable objects, sweeps dead ones, then **compacts** memory to eliminate fragmentation. |

<Callout type="warning" title="Key insight">
  Garbage collection runs **behind the scenes** using the Mark and Sweep
  algorithm. You as a developer never manage memory manually (unlike
  C/C++). But understanding GC helps you write code that doesn't
  accidentally keep references to huge objects, causing memory leaks.
</Callout>

## Other JS engines — V8 isn't the only one

The PDF notes that all these processes (parsing, interpreting, compiling,
GC) work **differently in each JavaScript engine**, but V8 is considered
the best on the market. Here's how other engines name their components:

| Engine             | Used in                 | Interpreter          | Compiler                |
| ------------------ | ----------------------- | -------------------- | ----------------------- |
| **V8**             | Chrome, Node.js, Deno   | Ignition             | TurboFan                |
| **SpiderMonkey**   | Firefox                 | Baseline Interpreter | WarpMonkey (IonMonkey)  |
| **JavaScriptCore** | Safari                  | LLInt                | DFG → FTL               |
| **Chakra**         | Old Edge (deprecated)   | Interpreter          | SimpleJIT → FullJIT     |

The architecture is the same concept everywhere: parse → interpret →
optimize hot code → deoptimize when assumptions fail. The names change,
the core idea doesn't.
