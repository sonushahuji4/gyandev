# `.pa11yci.json` — configuration notes

## Why `hideElements: "[aria-hidden='true']"`

Decorative icon glyphs (e.g. `<span aria-hidden="true">☰</span>` inside a button
that has its own `aria-label`) are not part of the accessibility tree — the
button's accessible name is its `aria-label`, and the glyph is visual only.

axe-core's `color-contrast` rule still evaluates the visible glyph. When the
glyph's computed background is a CSS custom property (`var(--color-bg)`) the
rule sometimes returns `needsFurtherReview: true` rather than a firm pass/fail.

- `@axe-core/cli` treats `needsFurtherReview` as **non-failing**.
- `pa11y-ci` treats `needsFurtherReview` as an **error**.

Rather than weakening `color-contrast` globally, we tell pa11y to skip every
`[aria-hidden="true"]` element. That matches how assistive tech already
ignores them, and leaves real text-contrast checks — body, links, headings,
button labels — fully enforced.

The token-level `scripts/check-contrast.mjs` gate (WCAG ratios computed from
`global.css` tokens) is the source-of-truth contrast check; axe-cli + pa11y
are runtime sanity checks.
