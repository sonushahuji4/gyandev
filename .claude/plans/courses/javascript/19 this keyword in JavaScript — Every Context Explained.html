this keyword in JavaScript — Every Context Explained

The value of this depends on HOW a function is called, not WHERE it's written — every scenario with exact examples

The Golden Rule — Before Everything Else

this is NOT determined by WHERE a function is written — it's determined by HOW it is called

This is the single most important thing to understand about this in JavaScript. The same function can have different this values depending on how you call it. This is what makes this confusing — and what interviewers always test.

Same function, different this values
function showThis() {
  console.log(this);
}

showThis();          // window (non-strict) or undefined (strict)
window.showThis();   // window — always, even in strict mode

const obj = { show: showThis };
obj.show();          // obj — because obj is before the dot

showThis.call({a:1}); // {a: 1} — explicitly set with call
Why does this exist? JavaScript functions are NOT attached to objects by default — they're standalone values. Unlike Java or C++ where methods always belong to a class, a JS function can be assigned to any variable, passed anywhere, called from any context. So this is the mechanism that tells the function: "here's the object you're working on right now." The same function can work with different objects just by changing how it's called.
1. this in Global Space

In global space, this = the global object

Anything defined globally is said to be in a global space. At the top level of your code (not inside any function), this always refers to the global object.

Global space
console.log(this); // window object (in browser)
Why does the global object differ? The global object depends on the runtime environment: In the browser, the global object is window. In Node.js, the global object is global. In web workers, the global object is self. The universal way to refer to it in any environment is globalThis (ES2020). But at the top level, this always points to whatever the global object is in that environment.
2. this Inside a Function

The value depends on strict mode vs non-strict mode

Function call — the behavior changes with strict mode
function x() {
  console.log(this);
}
x();

// In NON-STRICT mode → window object
// In STRICT mode ("use strict") → undefined
Why does this feel confusing? On first look, it seems like this in global space and this inside a function behave the same (both show window in non-strict mode). But in reality they are different. The this inside a function is actually undefined. The reason it shows window in non-strict mode is because of a mechanism called this substitution.

this substitution — The Hidden Mechanism

According to this substitution: if the value of this keyword is null or undefined, it will be replaced by the globalObject — but only in non-strict mode.

Here's what actually happens step by step:

  1. You call x() — a standalone function call
  2. JS determines this = undefined (because there's no object before the dot)
  3. Non-strict mode: JS sees this is undefined → replaces it with window (this substitution)
  4. Strict mode: JS sees this is undefined → leaves it as undefined (no substitution)
To summarize: The actual value of this inside a function is undefined. But because of this substitution, in non-strict mode it gets replaced by window. In strict mode, it stays undefined. This is why strict mode is considered safer — it shows you the real value instead of hiding it behind substitution.

this depends on HOW the function is called — proof

Same function, different call style — different this
"use strict";

function x() {
  console.log(this);
}

x();         // undefined — standalone call, no object
window.x(); // window — called as a method of window object!
Why is this different? When you call x(), there's no object before the dot — it's a standalone call. In strict mode, this = undefined. But window.x() is a method call — the object before the dot is window. So this = window. This proves that this depends on how the function is called, not where it's written. Same function, two different this values.
3. this Inside an Object's Method

When a function is called as a method of an object, this = that object

A function that is a property of an object is called a method. When you call it using obj.method(), this inside the method refers to the object before the dot.

this inside a method — refers to the calling object
const obj = {
  a: 10,
  x: function() {
    console.log(this);   // {a: 10, x: ƒ}
    console.log(this.a); // 10
  },
};

obj.x(); // this = obj — because obj is the object before the dot
Why this = obj? The rule is simple: the object that appears before the dot in the function call becomes this. obj.x() → this = obj. This is called implicit binding — JS implicitly sets this to the calling object. This works for any depth: a.b.c.method() → this = c (the immediate object before the method).

The "Lost this" Problem — Extracting a Method

What happens when you copy a method to a variable?
const obj = {
  a: 10,
  x: function() {
    console.log(this.a);
  },
};

obj.x();          // 10 ✓ — method call, this = obj

const fn = obj.x;  // extracting the function reference
fn();              // undefined ✗ — standalone call, this = window (no 'a' on window)
Why does extracting break this? When you write const fn = obj.x, you're copying the function reference — NOT the object. fn is now a standalone function with no object attached. When you call fn(), there's no object before the dot, so this falls to the default rule (window in non-strict, undefined in strict). The function has "lost" its connection to obj. This is the exact problem that call, apply, and bind solve.
4. this with call, apply & bind

Explicitly setting this — method borrowing

call, apply, and bind let you explicitly tell a function what this should be. This is called explicit binding — you're overriding the default behavior.

Method borrowing with call — from the episode
const student = {
  name: "Alok",
  printName: function() {
    console.log(this.name);
  },
};

student.printName(); // "Alok" — this = student (implicit binding)

const student2 = {
  name: "Kajal",
};

// student2 doesn't have printName method!
// student2.printName(); → TypeError: not a function

// Solution: BORROW the method using call
student.printName.call(student2);
// "Kajal" — this is now student2 instead of student!
What's happening here? student.printName.call(student2) says: "Take the printName function from student, but run it with this set to student2." Inside printName, this.name now looks up student2.name → "Kajal". This is method borrowing — reusing one object's method for another object without duplicating code. call, apply, and bind are all used to set the value of this keyword.

Quick Reminder — call vs apply vs bind

  • fn.call(obj, arg1, arg2) — calls immediately, args one by one
  • fn.apply(obj, [arg1, arg2]) — calls immediately, args as array
  • const newFn = fn.bind(obj) — does NOT call. Returns new function with this permanently bound
5. this Inside Arrow Function — The Most Tricky Part

Arrow functions do NOT have their own this

This is the most important thing to understand about arrow functions: they do not have their own this value. Instead, they take this from the enclosing lexical context — meaning the scope where the arrow function is physically written.

Example 1: Arrow function directly inside an object — TRAP!

Arrow as a method — this is NOT what you expect
const obj = {
  a: 10,
  x: () => {
    console.log(this); // window! NOT obj!
  },
};

obj.x();
OUTPUTwindow object (NOT {a: 10, x: ƒ})
Why is this window and not obj? Arrow functions don't have their own this — they take it from the enclosing lexical context. Where is this arrow function physically written? Inside the obj object literal. But an object literal is NOT a scope — it doesn't create its own execution context. So the enclosing lexical context of this arrow function is the global scope. In the global scope, this = window. That's why this inside the arrow function is window, not obj.
This is a common interview trap! Many developers expect this inside an arrow function in an object to be the object. It's NOT. An object literal { } is NOT a scope. It's just a data structure. Only functions, blocks, and the global scope create scoping contexts. So the arrow function looks "past" the object literal to find its enclosing scope — which is the global scope.

Example 2: Arrow function inside a regular method — this WORKS correctly

Arrow function inherits this from the enclosing function
const obj2 = {
  a: 10,
  x: function() {
    // Regular function x — has its own this
    // When called as obj2.x(), this = obj2

    const y = () => {
      console.log(this);
      // Arrow function y — no own this
      // Takes this from enclosing lexical context
      // Enclosing context = function x
      // x's this = obj2
      // Therefore y's this = obj2 ✓
    };
    y();
  },
};

obj2.x();
OUTPUT{a: 10, x: ƒ} — this is obj2 ✓
Why does this work? The arrow function y is physically written inside the regular function x. x is a regular function — it HAS its own this. When called as obj2.x(), x's this = obj2. The arrow function y inherits this from its enclosing lexical context, which is x. So y's this = obj2. This is exactly WHY arrow functions were created — to solve the "nested function loses this" problem that existed before ES6.

The Problem Arrow Functions Were Created to Solve

Before arrow functions — nested function lost this
const obj = {
  a: 10,
  x: function() {
    console.log(this.a); // 10 ✓ — method call

    function inner() {
      console.log(this.a); // undefined ✗ — standalone call!
    }
    inner();

    // Old fix before arrow functions:
    var self = this; // save this in a variable
    function inner2() {
      console.log(self.a); // 10 ✓ — using saved reference
    }
    inner2();
  },
};
obj.x();
After arrow functions — clean solution
const obj = {
  a: 10,
  x: function() {
    console.log(this.a); // 10 ✓

    const inner = () => {
      console.log(this.a); // 10 ✓ — inherits from x!
    };
    inner();
  },
};
obj.x();
Arrow functions were created to solve this exact problem. Before ES6, nested functions inside methods lost this, and developers had to use hacks like var self = this or .bind(this). Arrow functions fix this elegantly by not having their own this — they automatically inherit it from the enclosing function.

Important: call/apply/bind CANNOT change an arrow function's this

Arrow function ignores explicit binding
const arrow = () => console.log(this);

arrow.call({ name: "test" });  // window — NOT {name:"test"}
arrow.apply({ name: "test" }); // window — ignores apply too
const bound = arrow.bind({ name: "test" });
bound(); // window — ignores bind too
Why can't call/apply/bind override arrow's this? Because arrow functions literally don't have a this binding slot. There's nothing to override. The this they use comes from their lexical scope — it's baked in at creation time and can never be changed. This is by design: arrow functions are meant to inherit this and that inheritance is permanent.
6. this Inside DOM Elements

In DOM event handlers, this = the HTML element

HTML
<button onclick="alert(this)">Click Me</button>
<!-- this = [object HTMLButtonElement] -->
<!-- this refers to the button element itself -->
JavaScript — addEventListener
document.getElementById("btn").addEventListener("click", function() {
  console.log(this); // the button element
});

// But with arrow function:
document.getElementById("btn").addEventListener("click", () => {
  console.log(this); // window! Arrow inherits from enclosing scope
});
Why does this differ between regular and arrow function in event handlers? With a regular function callback, the browser sets this to the element that triggered the event. With an arrow function, the browser can't set this (arrow functions don't have their own) — so it inherits from the enclosing lexical scope, which is usually the global scope (window). Use regular functions for event handlers if you need access to the element via this. Or use event.target / event.currentTarget instead.
7. this Inside a Class (Bonus — Not in Episode)

In a class constructor and methods, this = the instance

this in ES6 classes
class User {
  constructor(name) {
    this.name = name;   // this = the new instance being created
  }

  greet() {
    console.log("Hello, " + this.name);
  }
}

const u = new User("Akshay");
u.greet(); // "Hello, Akshay" — this = u (the instance)

// BUT: extracting the method loses this (same problem!)
const fn = u.greet;
fn(); // TypeError in strict mode (classes are always strict)
Why classes use strict mode: Class bodies run in strict mode automatically. So extracting a method gives this = undefined (not window), which causes a TypeError when trying to access undefined.name. This is why React class components needed this.handleClick = this.handleClick.bind(this) in the constructor.
8. this Inside setTimeout (Bonus — Interview Favorite)

setTimeout's callback loses this — unless you handle it

The problem and 3 solutions
const user = {
  name: "Akshay",
  greet: function() {
    console.log("Hello, " + this.name);
  }
};

// ❌ Problem: loses this
setTimeout(user.greet, 1000); // "Hello, undefined"

// ✅ Fix 1: bind
setTimeout(user.greet.bind(user), 1000); // "Hello, Akshay"

// ✅ Fix 2: arrow function wrapper
setTimeout(() => user.greet(), 1000); // "Hello, Akshay"

// ✅ Fix 3: self reference
const obj = {
  name: "Akshay",
  greet: function() {
    const self = this;
    setTimeout(function() {
      console.log("Hello, " + self.name); // uses closure
    }, 1000);
  }
};
obj.greet(); // "Hello, Akshay"
Why does setTimeout lose this? When you write setTimeout(user.greet, 1000), you're extracting the function reference and passing it to the browser's timer. The browser stores it and calls it later as a standalone function — no object before the dot. So this defaults to window. bind fixes this by creating a new function with this permanently locked. The arrow wrapper keeps it as a method call (user.greet() — object before the dot).

Interview Questions & Answers
Q1What is the this keyword in JavaScript?
Answer

this is a special keyword that refers to an object. Which object it refers to depends on how the function is called, not where it's written. In global space, this = the global object (window in browsers). In a method call (obj.fn()), this = the object before the dot. In a standalone call (fn()), this = window (non-strict) or undefined (strict). With call/apply/bind, this = the explicitly provided object. In arrow functions, this = inherited from enclosing lexical scope.

Q2What is "this substitution"?
Answer

This substitution is a behavior in non-strict mode where if the value of this is null or undefined, JavaScript automatically replaces it with the global object (window in browsers). This is why this inside a standalone function call shows window in non-strict mode — the actual value is undefined, but this substitution replaces it. In strict mode, no substitution happens — this stays as undefined. This is one reason strict mode is considered safer.

Q3Why does this inside an arrow function in an object literal point to window, not the object?
Answer

Arrow functions don't have their own this — they inherit it from the enclosing lexical context. An object literal { } is NOT a scope — it doesn't create an execution context. So when an arrow function is directly inside an object literal, its enclosing lexical context is the scope where the object is defined — which is typically the global scope. So this = window. To get this = the object, put the arrow function inside a regular function method — the regular function has its own this (set to the object), and the arrow function inherits it.

Q4Can you change an arrow function's this using call, apply, or bind?
Answer

No. Arrow functions don't have a this binding slot — there's nothing to override. call, apply, and bind can pass arguments to an arrow function, but the this value is always ignored — it remains whatever it inherited from the enclosing lexical scope at creation time. This is by design: arrow functions were created specifically to have a permanent, inherited this.

Q5What is the priority order of this binding rules?
Answer

From highest to lowest priority: (1) new keyword — this = newly created object. (2) Explicit binding — call/apply/bind set this to the provided object. (3) Implicit binding — obj.fn() sets this = obj. (4) Default binding — standalone fn() sets this = window (non-strict) or undefined (strict). (5) Arrow functions — no own this, inherits from enclosing scope, cannot be overridden. The new keyword can even override bind, but nothing can override an arrow function's this.

Q6Why does setTimeout lose this and how do you fix it?
Answer

When you pass a method to setTimeout (setTimeout(obj.greet, 1000)), you're extracting the function reference and handing it to the browser timer. The browser calls it later as a standalone function — no object before the dot — so this defaults to window. Three fixes: (1) setTimeout(obj.greet.bind(obj), 1000) — bind permanently locks this. (2) setTimeout(() => obj.greet(), 1000) — arrow function preserves the method call syntax. (3) Save this in a variable: const self = this and use self in the callback (old-school approach).

Quick Revision — Cheat Sheet

this keyword — Everything to Remember

  • this depends on how a function is called, NOT where it's written.
  • Global space: this = window (browser) / global (Node.js) / globalThis (universal).
  • Function call f(): this = undefined. But in non-strict mode, this substitution replaces it with window.
  • Method call obj.f(): this = obj (the object before the dot). Implicit binding.
  • Nested object a.b.f(): this = b (immediate object before the dot, not a).
  • call/apply/bind: Explicitly sets this. call = comma args. apply = array. bind = returns new function.
  • new Fn(): this = new empty object. Highest priority.
  • Arrow function: NO own this. Inherits from enclosing lexical context. CANNOT be changed by call/apply/bind.
  • Arrow in object literal: this = window/global (object is NOT a scope!).
  • Arrow inside regular method: this = method's this (works correctly!).
  • DOM handler (regular fn): this = the HTML element.
  • DOM handler (arrow fn): this = enclosing scope (usually window).
  • setTimeout(obj.fn): Loses this. Fix with .bind(obj) or arrow wrapper.
  • Extracting method: const fn = obj.method; fn() → loses this.
  • Priority order: new > call/apply/bind > obj.fn() > fn() standalone > arrow (inherits).
  • bind is permanent: call/apply/second bind can't override. Only new overrides bind.
  • this substitution: null/undefined → replaced by window in non-strict mode only.
  • Classes are always strict mode → extracting method gives undefined, not window.
