Episode 23 — What is async?

async makes a function return a promise — always

The async keyword before a function does one simple thing: it ensures the function always returns a promise. Even if you return a plain string or number, async wraps it in a promise automatically.

Example 1: Returning a string from async function

async wraps the return value in Promise.resolve()
async function getData() {
  return "Namaste JavaScript";
}

const dataPromise = getData();
console.log(dataPromise);
// Promise {<fulfilled>: 'Namaste JavaScript'}

// How to extract the actual value? Use .then:
dataPromise.then(res => console.log(res));
// "Namaste JavaScript"
Why does async wrap the value? Because the whole point of async functions is to work with the promise ecosystem. By always returning a promise, async functions can be chained with .then(), composed with Promise.all(), and consumed consistently. Even if your function is synchronous inside, the caller always gets a promise — making the API predictable.

Example 2: Returning an existing promise — no double wrapping

async is smart — it doesn't wrap a promise inside another promise
const p = new Promise((resolve, reject) => {
  resolve("Promise resolved value!!");
});

async function getData() {
  return p; // returning an existing promise
}

const dataPromise = getData();
console.log(dataPromise);
// Promise {<fulfilled>: 'Promise resolved value!!'}
// NOT Promise { Promise { ... } } — no double wrapping!

dataPromise.then(res => console.log(res));
// "Promise resolved value!!"
Why no double wrapping? JavaScript checks: is the returned value already a "thenable" (an object with a .then method)? If yes, it returns it as-is. If no, it wraps it with Promise.resolve(). This prevents the confusing "promise inside a promise" situation and keeps the API clean.
What is await? — How We Used to Handle Promises vs Now

The old way — Promise.then/.catch

Before async/await — we used .then
const p = new Promise((resolve, reject) => {
  resolve("Promise resolved value!!");
});

function getData() {
  p.then(res => console.log(res));
}

getData(); // "Promise resolved value!!"

The new way — async/await

await replaces .then — reads like synchronous code
async function handlePromise() {
  const val = await p;
  console.log(val);
}

handlePromise(); // "Promise resolved value!!"

await can only be used inside an async function. Using it outside throws: SyntaxError: await is only valid in async function.

What Makes async/await Special? — The Key Difference

The comparison that matters — adding a delay to the promise

Promise that takes 3 seconds to resolve
const p = new Promise((resolve, reject) => {
  setTimeout(() => {
    resolve("Promise resolved value!!");
  }, 3000);
});

Promise.then way

JS does NOT wait
function getData() {
  p.then(res => console.log(res));
  console.log("Hello There!");
}
getData();
OUTPUT"Hello There!" → (3s) → "Promise resolved value!!"

JS registers the .then callback and moves on immediately. "Hello There!" prints first. After 3 seconds, the promise resolves and the .then callback runs.

async/await way

Function WAITS at await line
async function handlePromise() {
  const val = await p;
  console.log("Hello There!");
  console.log(val);
}
handlePromise();
OUTPUT(3s wait) → "Hello There!" → "Promise resolved value!!"

The function pauses at await until the promise resolves. Then continues line by line. Both lines print together after 3 seconds.

This is THE major difference: With .then, JS doesn't wait — it registers the callback and moves on, so code runs out of order. With await, the function pauses at that line and waits for the promise to resolve before moving to the next line. The code reads top-to-bottom, in order — like synchronous code. This is what makes async/await so much more readable.
Deep Dive — Multiple awaits & Timing Scenarios

Scenario 1: Two awaits on the SAME promise (p takes 3s)

Will it wait 3+3 = 6 seconds? Or just 3?
async function handlePromise() {
  console.log("Hi");

  const val = await p;     // first await on p
  console.log("Hello There!");
  console.log(val);

  const val2 = await p;    // second await on SAME p
  console.log("Hello There! 2");
  console.log(val2);
}
handlePromise();
OUTPUT"Hi" → (3s) → "Hello There!" → "Promise resolved value!!" → "Hello There! 2" → "Promise resolved value!!"
Why only 3 seconds total, not 6? Both awaits reference the same promise p. The promise was created once, its timer started once. After 3 seconds it resolves. The first await gets the value. The second await gets it instantly because p is already resolved — a settled promise returns immediately. You don't create a new timer each time you await.

Scenario 2: Two DIFFERENT promises — p (3s) and p2 (2s)

await p first, then await p2
const p = new Promise(resolve =>
  setTimeout(() => resolve("Promise resolved value!!"), 3000)
);
const p2 = new Promise(resolve =>
  setTimeout(() => resolve("Promise resolved value by p2!!"), 2000)
);

async function handlePromise() {
  console.log("Hi");

  const val = await p;       // waits for p (3s)
  console.log("Hello There!");
  console.log(val);

  const val2 = await p2;     // p2 already resolved at 2s!
  console.log("Hello There! 2");
  console.log(val2);
}
handlePromise();
OUTPUT"Hi" → (3s) → all 4 lines print together instantly
Why does p2 print instantly after p? Both promises were created at the same time — both setTimeout timers started ticking immediately. p2's 2-second timer finished at the 2-second mark. p's 3-second timer finished at the 3-second mark. We await p first (waits 3s). By then, p2 has been resolved for 1 second already. So await p2 returns immediately. Promises start at creation, not at await!

Scenario 3: Same promises, REVERSED order — await p2 first, then p

What changes when we swap the order?
async function handlePromise() {
  console.log("Hi");

  const val = await p2;      // p2 takes 2s
  console.log("Hello There!");
  console.log(val);

  const val2 = await p;      // p takes 3s (but 2s already passed!)
  console.log("Hello There! 2");
  console.log(val2);
}
handlePromise();
OUTPUT"Hi" → (2s) → "Hello There!" + p2 value → (1s more) → "Hello There! 2" + p value
Why the timing splits this time: We await p2 first (2s timer). After 2 seconds, p2 resolves → first two lines print. Then we await p. p's timer was 3 seconds but started at the same time as p2. At this point 2 seconds have already passed, so p only needs 1 more second. After that 1 second, p resolves and the remaining lines print. Total: 3 seconds, but output comes in two batches.

Scenario 4: p1 = 10s, p2 = 5s — The Big Question

await p1 first (10s), then p2 (5s) — what happens?
const p1 = new Promise(resolve =>
  setTimeout(() => resolve("Promise resolved value by p1!!"), 10000)
);
const p2 = new Promise(resolve =>
  setTimeout(() => resolve("Promise resolved value by p2!!"), 5000)
);

async function handlePromise() {
  console.log("Hi");

  const val = await p1;      // waits for p1 (10s)
  console.log("Hello There!");
  console.log(val);

  const val2 = await p2;     // p2 resolved at 5s — already done!
  console.log("Hello There! 2");
  console.log(val2);
}
handlePromise();
OUTPUT"Hi" → (10s) → "Hello There!" + p1 val → "Hello There! 2" + p2 val (instant!)
Why p2 resolves instantly after p1 even though it's 5 seconds? Because p2's timer started at the SAME TIME as p1 (when the const p2 = ... line executed). p2's 5-second timer finished at the 5-second mark. We were stuck waiting for p1 (10s). When p1 finally resolves at 10s, p2 has been resolved for 5 seconds already. So await p2 returns immediately. Key rule: await does NOT start a promise. It only waits for an already-running promise.

How to Make p2 Actually Start AFTER p1 Finishes

Use factory functions — create the promise INSIDE the async function
function createP1() {
  return new Promise(resolve =>
    setTimeout(() => resolve("p1 resolved"), 10000)
  );
}
function createP2() {
  return new Promise(resolve =>
    setTimeout(() => resolve("p2 resolved"), 5000)
  );
}

async function handlePromise() {
  console.log("Hi");

  const val1 = await createP1(); // p1 CREATED here, timer starts now
  console.log(val1);              // "p1 resolved" at 10s

  const val2 = await createP2(); // p2 CREATED here, timer starts NOW (at 10s)
  console.log(val2);              // "p2 resolved" at 15s (10 + 5)
}
handlePromise();
OUTPUT"Hi" → (10s) → "p1 resolved" → (5s more) → "p2 resolved"
Why this works: The promise is NOT created upfront. createP2() is called only AFTER await createP1() finishes. So p2's timer starts at the 10-second mark. It takes 5 more seconds → total 15 seconds. This is the pattern for truly sequential async operations where the second operation should only begin after the first completes.
Behind the Scenes — Call Stack Mechanics

JS is NOT actually waiting — the function is SUSPENDED

"Time, Tide, and JS wait for none." The call stack is never blocked. Here's what happens step by step:

The critical insight: When JS hits await, it does NOT freeze the engine. It suspends only that function and pops it off the call stack. Real-World Example & Error Handling

Fetching data with async/await

The pattern you'll use daily — fetch needs TWO awaits
async function handlePromise() {
  // fetch() returns a Response object (wrapped in a promise)
  const data = await fetch("https://api.github.com/users/alok722");
  // data is now a Response object, body is a ReadableStream

  // response.json() is ALSO a promise — parses the body as JSON
  const res = await data.json();
  console.log(res);
}
handlePromise();
Why two awaits for fetch? (1) fetch() resolves when HTTP headers arrive → gives a Response object. (2) response.json() resolves when the entire body is received and parsed → gives the actual data. Two async steps = two awaits. Same reason you'd write fetch(url).then(r => r.json()).then(data => ...) with promises.

Error Handling — Two Ways

Way 1: try/catch (recommended)
async function handlePromise() {
  try {
    const data = await fetch("https://api.github.com/users/alok722");
    const res = await data.json();
    console.log(res);
  } catch (err) {
    console.log(err);
    // Catches network errors, bad URL, JSON parse errors
  }
}
handlePromise();
Way 2: .catch() on the async function call
async function handlePromise() {
  const data = await fetch("https://api.github.com/users/alok722");
  const res = await data.json();
  console.log(res);
}

// Since async functions return promises, .catch() works too:
handlePromise().catch(err => console.log(err));

async/await vs Promise.then/.catch — Which to Use?

They are the same thing underneath. async/await is syntactic sugar over promises. Behind the scenes, it uses the exact same Promise mechanism. The difference is purely in readability:

  • async/await solves shortcomings of Promise chaining — no more nested .then chains
  • async/await increases readability — code reads top-to-bottom like synchronous code
  • async/await has cleaner error handling with try/catch
  • It is always advisable to use async/await for new code

Episode 24 — Promise APIs: all, allSettled, race, any

4 ways to handle multiple promises together — with every scenario explained

One simply doesn't use async/await without knowing promises! These 4 APIs are essential for handling parallel async operations. Each behaves differently on success and failure — and interviewers love testing the differences.
Setup — Common Promises for All Scenarios

Our base scenario: p1 = 3s, p2 = 1s, p3 = 2s

For each API, we'll test two scenarios: (1) all succeed, (2) one or more fail. This is exactly how interviewers test you — "what if p2 fails?"

Promise.all() — All Must Succeed or Everything Fails

What it does

Promise.all([p1, p2, p3]) takes an array of promises and returns a single promise that:

  • Resolves when ALL input promises resolve → returns array of all results in input order
  • Rejects immediately on the FIRST rejection → doesn't wait for others

It's a fail-fast API. One failure = everything fails.

Scenario 1: All 3 succeed ✅

p1=3s success, p2=1s success, p3=2s success
const p1 = new Promise(resolve => setTimeout(() => resolve("P1 Success"), 3000));
const p2 = new Promise(resolve => setTimeout(() => resolve("P2 Success"), 1000));
const p3 = new Promise(resolve => setTimeout(() => resolve("P3 Success"), 2000));

Promise.all([p1, p2, p3]).then(results => {
  console.log(results);
});
OUTPUT (after 3s)['P1 Success', 'P2 Success', 'P3 Success']
Waits for the slowest (p1 = 3s). Results are in INPUT order, not resolution order. Even though p2 resolved first (1s), it's at index 1 matching the input array.

Scenario 2: p2 fails at 1s ❌

p1=3s success, p2=1s REJECT, p3=2s success
const p1 = new Promise(resolve => setTimeout(() => resolve("P1 Success"), 3000));
const p2 = new Promise((_, reject) => setTimeout(() => reject("P2 Fail"), 1000));
const p3 = new Promise(resolve => setTimeout(() => resolve("P3 Success"), 2000));

Promise.all([p1, p2, p3])
  .then(results => console.log(results))
  .catch(err => console.log(err));
OUTPUT (after 1s)"P2 Fail"
Rejects IMMEDIATELY at 1s when p2 fails. Does NOT wait for p1 and p3. Note: p1 and p3 are NOT cancelled — their timers keep running — but Promise.all ignores their results.
When to use Promise.all: When you need data from ALL sources and one failure means you can't proceed. Example: loading a dashboard that needs user data + notifications + settings. If any API fails, the dashboard can't render — so fail fast and show an error.
Promise.allSettled() — Wait for All, Never Fails

What it does

Promise.allSettled([p1, p2, p3]) waits for ALL promises to settle (success or failure) and returns an array of objects describing each outcome. It never rejects.

Scenario 1: All 3 succeed ✅

Same as Promise.all — waits for slowest (3s), returns all results. But format is different:

OUTPUT (after 3s)[{status:'fulfilled',value:'P1 Success'}, {status:'fulfilled',value:'P2 Success'}, {status:'fulfilled',value:'P3 Success'}]

Scenario 2: p3 fails at 2s — allSettled STILL waits for all

p1=3s success, p2=1s success, p3=2s REJECT
const p1 = new Promise(resolve => setTimeout(() => resolve("P1 Success"), 3000));
const p2 = new Promise(resolve => setTimeout(() => resolve("P2 Success"), 1000));
const p3 = new Promise((_, reject) => setTimeout(() => reject("P3 Fail"), 2000));

Promise.allSettled([p1, p2, p3])
  .then(results => console.log(results))
  .catch(err => console.log(err)); // .catch will NOT trigger!
OUTPUT (after 3s)[{status:'fulfilled',value:'P1 Success'}, {status:'fulfilled',value:'P2 Success'}, {status:'rejected',reason:'P3 Fail'}]
Waits for ALL to settle (3s for p1, the slowest). p3's failure doesn't short-circuit. Each result has status + value (fulfilled) or status + reason (rejected).
Promise.all = Fail Fast. Promise.allSettled = Wait and provide accumulative result. allSettled is the safest Promise API — it always gives you results for every promise.
When to use Promise.allSettled: When you want results of ALL operations regardless of individual failures. Example: sending emails to 100 users — some might fail, but you need to know which succeeded and which failed. Unlike Promise.all, one failure doesn't hide the successes.
Promise.race() — First to Settle Wins (Success OR Failure)

What it does

Promise.race([p1, p2, p3]) returns the result of the first promise to settle — whether it's a success or failure. The "race" is about speed: whoever finishes first, their result (or error) becomes the output.

Scenario 1: p2 resolves first at 1s ✅

p1=3s success, p2=1s success, p3=2s reject
const p1 = new Promise(resolve => setTimeout(() => resolve("P1 Success"), 3000));
const p2 = new Promise(resolve => setTimeout(() => resolve("P2 Success"), 1000));
const p3 = new Promise((_, reject) => setTimeout(() => reject("P3 Fail"), 2000));

Promise.race([p1, p2, p3])
  .then(result => console.log(result))
  .catch(err => console.log(err));
OUTPUT (after 1s)"P2 Success"
p2 settles first (1s, success) → that's the result. p1 and p3 still finish but race doesn't care.

Scenario 2: p3 REJECTS first at 2s ❌

p1=3s success, p2=5s success, p3=2s REJECT
const p1 = new Promise(resolve => setTimeout(() => resolve("P1 Success"), 3000));
const p2 = new Promise(resolve => setTimeout(() => resolve("P2 Success"), 5000));
const p3 = new Promise((_, reject) => setTimeout(() => reject("P3 Fail"), 2000));

Promise.race([p1, p2, p3])
  .then(result => console.log(result))
  .catch(err => console.log(err));
OUTPUT (after 2s)"P3 Fail"
p3 settles first (2s, rejection) → race returns the rejection. Even though p1 would succeed at 3s, race already settled.
When to use Promise.race: Implementing timeouts. Race your fetch against a setTimeout that rejects after 3 seconds. Whoever settles first wins — if the fetch is slow, the timeout wins and you show an error. Also useful for selecting the fastest server response.
Promise.any() — First SUCCESS Wins (Failures Ignored)

What it does

Promise.any([p1, p2, p3]) waits for the first promise to SUCCEED. Rejections are ignored until one succeeds. Only if ALL fail does it reject with an AggregateError.

Scenario 1: p3 fails at 2s but p1 succeeds at 3s ✅

p1=3s success, p2=5s success, p3=2s REJECT
const p1 = new Promise(resolve => setTimeout(() => resolve("P1 Success"), 3000));
const p2 = new Promise(resolve => setTimeout(() => resolve("P2 Success"), 5000));
const p3 = new Promise((_, reject) => setTimeout(() => reject("P3 Fail"), 2000));

Promise.any([p1, p2, p3])
  .then(result => console.log(result))
  .catch(err => console.log(err));
OUTPUT (after 3s)"P1 Success"
p3 fails at 2s — ignored! Promise.any keeps waiting for a success. p1 succeeds at 3s — that's the result.

Scenario 2: Two fail, one succeeds ✅

p1=3s REJECT, p2=5s success, p3=2s REJECT
const p1 = new Promise((_, reject) => setTimeout(() => reject("P1 Fail"), 3000));
const p2 = new Promise(resolve => setTimeout(() => resolve("P2 Success"), 5000));
const p3 = new Promise((_, reject) => setTimeout(() => reject("P3 Fail"), 2000));

Promise.any([p1, p2, p3])
  .then(result => console.log(result))
  .catch(err => console.log(err));
OUTPUT (after 5s)"P2 Success"
p3 fails at 2s — ignored. p1 fails at 3s — ignored. p2 succeeds at 5s — that's the result! Promise.any only cares about successes.

Scenario 3: ALL three fail ❌

p1=3s REJECT, p2=5s REJECT, p3=2s REJECT
const p1 = new Promise((_, reject) => setTimeout(() => reject("P1 Fail"), 3000));
const p2 = new Promise((_, reject) => setTimeout(() => reject("P2 Fail"), 5000));
const p3 = new Promise((_, reject) => setTimeout(() => reject("P3 Fail"), 2000));

Promise.any([p1, p2, p3])
  .then(result => console.log(result))
  .catch(err => {
    console.log(err);         // AggregateError: All promises were rejected
    console.log(err.errors);  // ['P1 Fail', 'P2 Fail', 'P3 Fail']
  });
OUTPUT (after 5s)AggregateError: All promises were rejected → errors: ['P1 Fail', 'P2 Fail', 'P3 Fail']
Waits for ALL to settle (5s, the slowest). Only when ALL fail does it reject with AggregateError. Use err.errors to get the individual rejection reasons.
When to use Promise.any: When you have fallbacks. Example: trying multiple CDN servers for a resource — you want the first one that works. Failures are expected (servers go down). Unlike race, a failure doesn't end the operation — it keeps looking for a success.

Summary of 6 static methods:
Promise.all(promises) — waits for all to resolve, fail-fast on first rejection
Promise.allSettled(promises) — waits for all to settle, returns status for each, never rejects
Promise.race(promises) — first to settle (success or failure) wins

Promise.reject(error) — creates a rejected promise with given error
Interview Questions & Answers
Q1What is an async function and how is it different from a normal function?
Answer

An async function is declared with the async keyword. Two differences: (1) It always returns a promise — if you return a plain value, it's wrapped in Promise.resolve(value); if you return a promise, it's returned as-is without double wrapping. (2) You can use await inside it. Normal functions cannot use await and don't automatically return promises.

Q2Does await actually block the call stack?
Answer

No! When JS hits await, it suspends the async function and pops it off the call stack. The stack is free for other code (UI updates, event handlers). When the awaited promise resolves, the event loop pushes the function back onto the stack and it resumes from where it left off. The page never freezes because the call stack is never blocked.

Q3Do promises start executing when created or when awaited?
Answer

Promises start executing immediately when created. The executor function runs synchronously at creation time. await doesn't start a promise — it only waits for an already-running one. If you create const p1 = new Promise(...) and const p2 = new Promise(...), both timers start immediately regardless of when you await them. To make them sequential, create them inside the async function using factory functions.

Q4What is the difference between Promise.all and Promise.allSettled?
Answer

Promise.all is fail-fast: it waits for all to succeed and returns results in input order. If ANY one rejects, it immediately rejects with that error without waiting for others. Promise.allSettled always waits for ALL promises to settle (success or failure) and returns an array of objects with {status, value/reason} for each. It never rejects. Use all when one failure means you can't proceed. Use allSettled when you need results of every operation regardless of failures.

Q5What is the difference between Promise.race and Promise.any?
Answer

Promise.race returns the first promise to settle — whether it's fulfilled or rejected. If the fastest promise fails, race returns that failure. Promise.any returns the first promise to succeed — rejections are ignored. It only rejects if ALL promises fail, giving an AggregateError with all rejection reasons in err.errors. Use race for timeouts (fastest settles). Use any for fallbacks (first success wins).

Q6What is AggregateError and when does it occur?
Answer

AggregateError is a special error type that wraps multiple errors into one. It occurs when Promise.any is used and ALL input promises are rejected. The error message is "All promises were rejected" and you can access individual rejection reasons via err.errors which returns an array of all rejection reasons. It was introduced in ES2021 along with Promise.any.

Q7In Promise.all, if one promise rejects, do the others get cancelled?
Answer

No! The other promises are NOT cancelled. They were already triggered and will continue running to completion (success or failure). But Promise.all ignores their results — it has already rejected. The promises themselves don't know about Promise.all; they're independent. This is an important distinction: Promise.all doesn't "stop" anything, it just stops waiting and immediately returns the first rejection.

Q8How do you handle errors in async/await?
Answer

Two approaches: (1) try/catch inside the async function (recommended) — wraps await calls, catches all rejections and thrown errors in the catch block. (2) .catch() on the function call — since async functions return promises, handlePromise().catch(err => ...) works. Both are valid because async/await is sugar over promises. try/catch is preferred for co-located, readable error handling.

Q9Why does fetch() need two awaits?
Answer

Because fetch involves two async steps: (1) await fetch(url) resolves when the HTTP headers arrive — gives a Response object. The body is still a ReadableStream at this point. (2) await response.json() resolves when the entire body is received and parsed as JSON — gives the actual data. Two async operations = two awaits. Same reason the .then chain has two steps: fetch(url).then(r => r.json()).then(data => ...).

Q10What is the difference between async/await and Promise.then/.catch?
Answer

They are the same thing underneath — async/await is syntactic sugar over promises. Behind the scenes, it uses the exact same Promise mechanism. Differences are purely in code style: async/await reads top-to-bottom like synchronous code (more readable), uses try/catch for error handling (cleaner), and is easier to debug. Promise.then is more explicit about what's async and better for patterns like Promise.all. Recommendation: Use async/await as default for new code. Use .then for Promise.all/race/any patterns.

  • Promise.allSettled — waits for ALL. Never rejects. Returns {status, value/reason} for each. Safest API.
  • Promise.race — first to SETTLE wins (success or failure). Use for timeouts.
  • Promise.any — first to SUCCEED wins. Ignores rejections. Only rejects if ALL fail → AggregateError. Use for fallbacks.
  • Promise.all rejects do NOT cancel other promises — they keep running but results are ignored.
  • Promise.resolve(value) — creates fulfilled promise. Promise.reject(error) — creates rejected promise.
  • Settled = fulfilled OR rejected. Once settled, state is permanent.
